## 1. 背景與問題陳述

### 1.1 現況描述

目前人物誌系統採用三層式設計：

| 層級 | 內容 | 題數 | 必填性 |
|-----|------|-----|--------|
| 第一層 | 基本資訊（開始年份、常去地點、路線類型） | 3 題 | 必填 |
| 第二層 | 核心故事（相遇、意義、建議） | 3 題 | 強烈建議 |
| 第三層 | 進階故事（6 大分類） | 31 題 | 選填 |

**總計：37 個欄位，其中 34 題為開放性文字填寫**

### 1.2 表面問題

用戶反映「填寫人物誌感覺很困難」，初步分析歸納出：

- 數量壓力（34 個開放性問題）
- 問題過於抽象/哲學性
- 純文字填寫門檻高
- 範例造成完美主義壓力
- 缺乏參考與引導

### 1.3 ⭐ 真正的問題（核心洞察）

經過更深入的用戶理解，發現**真正的障礙不是「填寫困難」，而是心理安全感不足**：

```
┌─────────────────────────────────────────────┐
│                                             │
│   「我才爬 V3，寫什麼人物誌？」               │
│   「我又沒去過龍洞，有什麼好分享的」          │
│   「萬一被強者看到笑我怎麼辦」                │
│                                             │
│   但是...                                   │
│                                             │
│   「好想看那個爬很強的人是怎麼開始的」        │
│   「想知道別人怎麼克服恐懼」                  │
│   「看別人的故事很有共鳴」                   │
│                                             │
└─────────────────────────────────────────────┘
```

**核心矛盾：用戶想「看」別人的故事，但不想「被看」**

| 想要 | 害怕 |
|-----|------|
| 被社群認同 | 被評判「不夠格」 |
| 記錄自己的故事 | 故事不夠精彩 |
| 連結其他岩友 | 暴露自己的程度 |

**心理學標籤：**
- 冒名頂替症候群 (Impostor Syndrome)
- 聚光燈效應 (Spotlight Effect)
- 社交比較焦慮 (Social Comparison Anxiety)

## 2. 策略方向

### 2.1 策略轉向

| 原策略 | 新策略 |
|-------|--------|
| 降低填寫門檻 | **建立心理安全感** |
| 讓問題更簡單 | **讓用戶覺得自己值得被記錄** |
| 增加引導和範例 | **正常化「普通」，去除成就暗示** |

### 2.2 核心訊息

> **「每個人的故事都值得被記錄，跟你爬多難沒關係」**

### 2.3 設計原則

1. **先看再寫** — 讓用戶先當觀眾，建立安全感
2. **私密優先** — 預設只有自己看得到，降低曝光焦慮
3. **去成就化** — 移除所有暗示「你要夠厲害」的語言
4. **正常化普通** — 刻意展示「普通人」的故事
5. **漸進式曝光** — 從私密 → 匿名 → 公開的自然過渡

## 3. 目標設定

### 3.1 核心目標

**讓用戶相信「我的故事值得被記錄」，進而願意分享**

### 3.2 量化指標

| 指標 | 現況（預估） | 目標 |
|-----|-------------|------|
| 開始填寫的用戶比例 | ~20% | 50% |
| 至少寫 1 則故事的比例 | ~15% | 40% |
| 從私密轉為公開的比例 | N/A（新功能） | 30% |
| 用戶瀏覽他人故事的比例 | ~40% | 70% |

### 3.3 質化目標

- 用戶回饋「原來不用很厲害也可以寫」
- 社群中出現更多「普通人」的故事
- 降低用戶對人物誌的焦慮感

## 4. 解決方案

### 4.0 ⭐ 問題分層重設計（核心改動）

#### 設計理念

將原本 34 個開放性文字題目，重新設計為**三層漸進式結構**：

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  第一層：點選 / Hashtag                    ← 零門檻         │
│  「選一選就完成了」                                         │
│                                                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  第二層：一句話系列                        ← 低門檻         │
│  「回答一句話就好」                                         │

│                                                             │
│  第三層：故事與思考                        ← 高門檻         │
│  「想分享完整故事的時候再來」                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.1 Phase 1：建立安全感基礎（1-2 週）

#### 4.1.1 新增隱私選項（P0）

**預設「僅自己可見」，降低曝光焦慮：**

```
你的故事要公開嗎？

○ 公開 — 所有人都能看到
○ 社群可見 — 只有登入的岩友能看
● 僅自己 — 先寫給自己看，之後再決定  ← 預設
○ 匿名分享 — 故事公開但不顯示名字
```

**影響檔案：**
- `src/lib/types.ts` — 新增 visibility 欄位
- `src/components/profile/PublicSettingSection.tsx`
- `backend/src/db/schema.sql` — 資料庫欄位

#### 4.1.2 修改產品定位文案（P0）

**頁面標題：**
```
之前：建立你的人物誌
之後：記錄你的攀岩故事
```

**空狀態文案：**
```
之前：還沒有填寫任何故事
之後：每個人都有故事值得記錄
      不管爬了多久、爬得多難
```

**CTA 按鈕：**
```
之前：開始填寫
之後：開始記錄（只有你看得到）
```

**影響檔案：**
- `src/app/profile/page.tsx`
- `src/components/profile/dashboard/` 相關組件
- `src/app/auth/profile-setup/` 相關頁面

#### 4.1.3 問題文案去成就化（P0）

**修改原則：移除所有暗示「你要夠厲害才能回答」的語言**

| 原問題 | 修改後 |
|-------|--------|
| 第一次完攀某個難度的感覺 | 有沒有一條路線讓你特別印象深刻？ |

### 4.4 Phase 4：社群與內容策略（持續進行）

#### 4.4.1 種子內容策略（P1）

**問題：** 初期沒有足夠的「普通人故事」可以展示

**解決方案：**
1. 團隊成員先寫 10-20 則不同背景的故事
2. 邀請 10-15 位不同程度的岩友參與 beta 測試
3. 確保有足夠的「新手」「一般人」故事

**內容多樣性目標：**
- 攀岩 < 1 年：至少 5 則
- 攀岩 1-3 年：至少 5 則
- 攀岩 > 3 年：至少 5 則
- 不同性別、年齡、職業背景

#### 4.4.2 故事精選策略（P2）

**精選標準調整：**

```
之前：展示最厲害的故事
之後：展示最能引起共鳴的故事
```

**精選維度：**
- 真實感（而非成就）
- 多樣性（不同背景）
- 共鳴度（按讚/留言數）

## 7. 風險與注意事項

| 風險 | 影響 | 緩解措施 |
|-----|-----|---------|
| 種子內容不足 | 「先看再寫」機制無法運作 | 團隊先產出內容、邀請 beta 用戶 |
| 過度強調「普通」導致社群失去吸引力 | 資深岩友流失 | 保持內容多樣性、不同 persona 有不同入口 |
| 隱私選項導致公開內容太少 | 社群互動不足 | 設計良好的漸進式曝光機制 |
| 文案改動影響 SEO | 搜尋流量下降 | 評估關鍵字影響、漸進式調整 |
